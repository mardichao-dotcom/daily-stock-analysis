## 題一:未來 6 個月最可能造成「停更」或「髒資料」的三個結構性弱點

已修事故(6/7 單檔卡死、6/12 連鎖逾時、6/15 起 19 天停更、7/6 白屏)全屬「**可用性**」故障,
且護欄已層層加固。下面三個弱點是「下一類」:護欄體系的盲區,而非既有 bug 的變體。

### W1(P0)K 線資料只驗「活性」,從不驗「有效性」——髒資料的主通道

**現狀**:tv_collect → import_kline 這條鏈上所有護欄(preflight、EVAL/INIT/單檔/全域 timeout、
連續失敗中止、三層白屏偵測、幽靈 bar 拒絕、>20% 略過閘、freshness watchdog)驗的都是
「有沒有回資料 / 有沒有前進」。**沒有任何一道驗「回來的 OHLC 數值對不對」**:

- `src/import_kline.py:69-93`:唯一的內容檢查是「日期不在未來」;數值層面 0 檢查
  (負值、零值、high<low、單日 ±50% 跳變、成交量異常都直接入庫)。
- `scripts/tv_collect.mjs:27-30`:資料取自 TradingView Desktop **私有壓縮 API**
  (`window.TradingViewApi._activeChartWidgetWV…mainSeries().bars()`),無任何合約保證。
- `src/import_kline.py:87`:P0-C 改的 `INSERT OR REPLACE` 是雙面刃——設計上讓收盤值覆寫半成品,
  但同樣讓**一批錯誤資料無條件覆寫既有正確歷史**(非台股每天重抓 3 根、增量模式最深可回寫 180 根,
  `scripts/tv_collect.mjs:118-121`)。

**觸發情境**(任一即成立):
1. TV Desktop 自動更新後私有 API 語意漂移——不是「壞掉」(preflight 會抓到),而是「回不同的數」:
   例如預設改為還原權息序列、symbol 解析規則改變、或登入 session 降級成延遲/免費資料
   (免費帳號部分交易所是 15 分延遲+盤中值)。bars() 照樣回「看起來合理」的數字,全鏈綠燈通過。
2. 週末/假日跑批時 TV 回快取或殘缺序列(6/14 的 6/15 幽靈 bar 就是此類,只是那次剛好日期在未來
   才被日期閘擋到;若幽靈 bar 帶的是「今天的錯誤值」,現有閘完全放行)。
3. REPLACE + refresh 把某天正確收盤覆寫成錯值後,**永遠不會自癒**(下次 refresh 只抓最近 3 根)。

**現有護欄為何擋不住**:全部護欄的判準是 liveness(有回應、有前進、比例合理),
一批「數值錯但格式對、日期對、數量對」的資料會 100% 通過 preflight、import、20% 閘、
freshness watchdog、verify_publish(它只驗 HTTP 200 與 JSON 結構,`src/verify_publish.py:84-96`),
直接進 kline.db → 計分 → standing 狀態機 → 發布。而 standing 狀態是**順序依賴**的(見 W2),
髒 K 棒觸發的錯誤站穩/取消寫進 state 後,連事後修正 K 線都救不回狀態。

**緩解方向**:
- import 前加數值 sanity 閘:high≥low≥0、|close−prev_close|/prev_close 超閾值(如 30%)進隔離區
  並告警,不直接 REPLACE 既有 row(「先隔離、隔日核可再覆寫」)。估 0.5 天。
- 每日抽 3–5 檔對第二來源(TWSE 官方收盤價 API)比對 close,誤差 >0.5% 告警。
  fetch_chips 已有 TWSE 官網 API 經驗可複用。估 0.5 天。
- 根治 = 換掉 TV/CDP 來源,見下方遷移評估。

### W2(P0)順序依賴的狀態機沒有決定性重建路徑——髒資料的放大器,兼停更引信

**現狀**:standing_state(kline.db)是全系統唯一「不可從原始資料一鍵重算」的狀態。
`evaluate_standing` 的結果依賴「每個交易日恰好評估一次、按時序、用當天的 key_prices」:

- **靜默缺口改寫結果**:單檔當日 K 棒缺漏(≤20% 閘內完全合法,現規模可容 26 檔/天)時,
  該檔 `score_one_symbol` 直接 return None(`src/run_filters_v2.py:702-704`),狀態機當天不推進。
  隔天資料回來,TRIGGERED 的 `days_since` 變 2 → 觸發**靜默作廢**(`src/triggers/standing.py:176-179`)。
  一檔本該站穩成立的訊號就這樣消失,無任何告警——Discord 只報「略過 N 檔」,不報「狀態機受影響」。
  6/11 體檢的 TWSE:3665 間歇缺 K 棒就是此類,當時只當顯示 bug 處理。
- **同日重跑非冪等**:對同一 date 重跑 run_filters_v2(手動修復、補跑常見),
  當天剛 STANDING 的線會走 `STANDING → MAINTAINING` 轉移(`standing.py:200-211`),
  `should_score` 從 True 變 False → **該檔總分變低、C 級「今天新成立」標籤消失、
  score_history 被 UPSERT 覆寫成錯的分數**(`score_history_io.write_batch`)。
  重跑越勤快,資料越髒。
- **重放無法還原歷史輸入**:6/15–7/3 停更後的補算(commit 8369116)靠手動按序重放,
  但 key_prices.json 是活檔案(朋友每週改),歷史某天的「當時的線」只存在於當日
  filtered_result snapshot,重放腳本讀的是**今天的** config——重放結果與「當時真跑」不等價。
- **停更引信**:key_prices.json 每週人工手繪轉檔(已有 6 個 .bak 修正輪次,含 overlap-fix、
  friend-rereview,說明輸入錯誤是常態)。`_score_one_line` 的 `float(price_str)`
  (`src/run_filters_v2.py:553`)與 `_validate_category` 的嚴格 ValueError
  (`src/scoring/given_price.py:45-51`)都未被 catch——**一條格式錯的線 = 整個 run_filters 崩潰
  = 當晚不發布**。朋友手滑一次,系統停更一晚;200 檔 × 每週 30 檔手繪,頻率只會上升。
  (prepare_charts 的 replay 有 try/except 容錯,`prepare_charts_v2.py:199-218`;filters 沒有——同輸入雙標準。)

**現有護欄為何擋不住**:20% 閘管「當天規模」,不管「單檔連續性」;freshness watchdog 管
DB 整體 MAX(date),不管 per-symbol 缺口;測試 538 綠管純函式正確性,不管「跨日執行紀律」。
狀態機的正確性假設(每日一次、按序、同輸入)沒有任何機制強制或驗證。

**緩解方向**:
- 把「評估日期」寫進 state(schema 加 `last_evaluated_date`),同日重跑直接 no-op(冪等),
  跨日缺口 ≥2 自動觸發該檔補放(逐日 replay 該 symbol)。估 1–1.5 天。
- run_filters 對單條線的解析/計分包 try/except:壞線隔離報告(Discord 列出哪檔哪條),
  不拖垮整跑。估 0.5 天。加 convert 階段 schema 驗證(價格可 float、區域 low<high、category 在白名單)。
- 每日 filtered_result 已含 key_prices_snapshot——補一支「用 snapshot 序列決定性重建 standing_state」
  的工具,讓狀態隨時可從快照重算,不再是孤本。估 1 天。

### W3(P1)git repo 當生產資料庫 + GitHub Pages 當唯一部署面——膨脹曲線與部署鏈的複合停更風險

**現狀與實測數據**:
- 每交易日 `docs/data/v2/{date}/` 寫入**完整快照**(非增量):135 檔 × 平均 44KB(indent=2
  pretty-print)= 6.0MB、~30 萬行,`publish.sh:47` `git add -f` 強制入 git(繞過 .gitignore)。
- publish.sh 的 30 天清理只刪工作樹;**git 歷史裡的 blob 永存**。目前 .git 8.1M 還小,
  但曲線是單調的:現規模 ≈ 1.5GB/年歷史增量,300 檔 ≈ **3.3GB/年**(13.2MB×250 交易日)。
  GitHub 有 soft limit(1GB 建議、Pages 站 1GB 硬限),一年內會撞。
- 部署鏈已出過兩次事:stage9 結案當晚 Pages build 數小時延遲(stage9_retrospective §4.5)、
  7/6 build 卡住需空 commit 重觸發(e4f9d3e)。`verify_publish` 的容忍窗只有 90s+90s
  (`src/verify_publish.py:340-355`),Pages build 時間隨 repo/站點膨脹只會更常超窗——
  屆時要嘛 publish 常態標 ❌(告警疲勞),要嘛人工養成「Pages 慢而已」的忽略習慣(告警失效)。
- **四個排程互相 push 同一 branch**(19:00 主跑、08:30 macro、05:30 us_refresh、週六 weekly),
  靠 `git pull --rebase --autostash` 硬扛(`run_macro.sh:44-48`,還得先 commit 才能 rebase,
  2c099f1 已為此修過一次)。寫入者越多、檔案越大,rebase 衝突/半套 push 的機率越高;
  autostash 對 state/ 未追蹤變更的處理在衝突時可能把工作樹留在中間態,下一輪排程直接 fail。

**觸發情境**:repo 過 GitHub 限制被限流/拒推(停更);Pages build 超過 verify 窗(告警噪音→真事故被淹沒);兩排程 rebase 撞車留下 rebase-in-progress 狀態(之後每一輪 git 操作全掛,直到人工介入)。

**現有護欄為何擋不住**:heartbeat/verify/supervisor 都在「單次 run」維度;膨脹是跨月曲線,
沒有任何監控項看 .git 大小、Pages build 時長、或 git 工作樹是否卡在 rebase 狀態。

**緩解方向**:
- chart JSON 去 indent(`prepare_charts_v2.py:315` `json.dump(..., indent=2)` → 無 indent,
  單檔 −40~50%,一行改動)+ 歷史快照只留線上需要的天數已做,再把 data 移到 **orphan branch
  單 commit force-push**(歷史永遠只有一份)或獨立 data repo。估 0.5–1.5 天。
- run 開頭加「git 工作樹健康檢查」(rebase/merge 殘留即告警中止)。估 0.25 天。
- 監控加兩個慢指標:.git 大小、publish→verify 通過耗時,週報列出。估 0.25 天。

## 題二:實作與既定決策的偏離

### D1(P0)區域計分完全未按 v2.2 §3-B「碰到當天 +N」——走的是兩天確認 + 私設「區間中點」門檻

- **規則**:`朋友規則_v2_2.md` §3-B「區域類:K 棒碰到當天 +N(v2.2 起區域不需要兩天確認,碰到即算)」;
  §1-D 觸發 = K 棒與區域交集。變更摘要表、對照表(觸發時機=K 棒交集當天、加分時機=**當天 +N**)三處一致。
- **實作**:`src/run_filters_v2.py:601-678` `_score_one_area` 把區域餵進線類站穩狀態機:
  碰到當天只轉 TRIGGERED(0 分),**隔天 open 與 close 都 ≥ 區間中點**才 STANDING(+N)。
  「中點」(`given_price = (low+high)/2`,行 622)不存在於任何規則文件。
- **實測復現**(隔離腳本,region [95,105]、weights 實檔):
  - Day1 K 棒交集且收在區內 → **score 0.0**(規則:+2)
  - Day2 開收都 >100(中點)→ score 2.0(晚一天,且多了規則沒有的門檻)
  - 情境 B:Day2 整天在區域內但低於中點(open 96/close 97)→ **兩天都 0 分,永不加分**
    (規則:Day1 就該 +2)
- **影響**:key_prices.json 現有 **145 條區域**(order_block/poc/fvg/gap)的加分全面時點錯誤或漏加,
  連帶分數、S/A/B 分級、C 級標籤、chart 事件標記。這不是邊角,是計分主幹的一塊。
- **佐證**:commit c7aa74e 自述「preserves first-touch +N semantics」與實際行為不符;
  `docs/現況快照_2026-06-02.md` 宣稱「§3-B 區域類碰到當天 +N(不需 2 天確認)」已完工——文件與 code 兩邊都以為做對了。
- **建議**:找朋友確認語意後改 `_score_one_area`:交集當天即 should_score(每次「新進入」算一次,
  需定義離開/再進入),區域不進兩天狀態機或狀態機僅用於「站穩中」顯示標籤。改動小(單函式),
  但需決定歷史分數是否重算。
