# AI 協作守則

> 這份文件給 Claude(以及任何未來協作的 AI)看,定義「什麼時候相信誰」。
> 用戶在 Project 知識庫上傳此檔,期望每次新對話 AI 都自動讀到。
> 最後更新:2026-06-02

---

## 業務規則討論場景

當對話涉及「站穩 / 跌破 / 計分 / 規則 / 級別 / 標籤 / 區域」等業務語意時:

- ❌ **禁止 web search**(不要查「教科書站穩定義」「業界做法」「TradingView 怎麼算 MACD」)
- ✅ **只用 Project 內的 `朋友規則_v2_2.md`**
- ✅ 不確定時 **問用戶**,不要憑訓練資料推測
- ✅ 引用規則時 **註明出處**(§1-D / §3-A / §3-B / §4 等)

理由:朋友規則是業務專屬定義,跟教科書版本不同(例:v2.2 站穩取消的「雙向 leave」
就不在任何 textbook 裡)。用 web 知識替代規則是 anti-pattern。

---

## 技術實作場景

當對話涉及「程式碼 / API / 平台 / 工具 / 命令」等技術問題時:

- ✅ **可以 web search**,但要 **引用具體連結 + 註明日期**
- ✅ 不確定時 **跑指令確認**(grep / curl / sqlite3 / python3 -c …)
- ✅ 看 `~/台股儀表板/src/` 程式碼確認當前實作
- ❌ 不準「我覺得是這樣」**憑記憶答 API**

理由:API 跟工具版本變化快,訓練資料容易過時。實際 `import` 失敗 / 跑出 error
就是直接證據。

## 重要規則優先順序

當訊息來源衝突時,依序採信:

```
1. 朋友規則_v2_2.md           ← 最高,業務邏輯唯一來源
2. stage8_spec.md /
   每日更新資料流架構_v1.md   ← 架構依據
3. Project 其他文件
4. 用戶當前對話
5. Web search                  ← 僅技術實作場景
6. AI 訓練記憶                 ← 最低,容易過時 / 教科書化
```

注意:**用戶當前對話 ≠ 用戶過往對話**。
過往對話可能已被 v2.2 等更新蓋過,引用舊對話前要先看 Project 文件確認還適用。

## 不確定時的行為

不確定時 **主動標明**,範例:

- 「我對這部分不確定,需要你確認」
- 「這個答案我有 X% 信心,建議用 Y 方式驗證」
- 「規則 §3-A 沒明確寫這個 case,要不要先回去問朋友?」

**絕對禁止**:

- ❌ 自信編一個聽起來合理的答案
- ❌ 用教科書知識替代用戶的規則
- ❌ 「應該是這樣吧」式的口語推測
- ❌ 自己定義一個 edge case 處理方式不告訴用戶

## 規則衝突時的處理

| 衝突類型 | 怎麼辦 |
|---|---|
| Project 文件 vs 對話內容 | 問用戶釐清,echo 兩邊差異 |
| v2.1 vs v2.2 | **永遠以 v2.2 為準**(v2.1 已 deprecated) |
| 文件 vs 程式碼 | 問用戶以哪邊為準;通常「規則文件是 source of truth,程式碼跟規則對齊」 |
| 用戶當前對話 vs Project 文件 | 假設用戶有最新 context,優先採信對話;但提醒文件未同步 |

## 給 Claude 的具體執行守則

### 開新對話時

1. 先讀 `~/台股儀表板/CLAUDE.md`(若存在)
2. 必要時讀 `朋友規則_v2_2.md`(業務問題)
3. 看 `docs/現況快照_*.md` 了解最新狀態
4. 跑 `git log --oneline -10` 看最近改了什麼

### 寫程式時

- 改 scoring / standing / 標籤邏輯前 → **先回去看 §3-A、§3-B**
- 改 render 前 → **看 §4 標籤體系**
- 新增模組前 → **看 §0 一句話總覽** 確認沒違反「找買點的雷達」定位

### Echo 給用戶時

- 引用規則時 **註明章節**(例:「依 §3-A 站穩取消條件 (b),這天會 CANCELLED」)
- 不確定時 **明說**(「以下 50% 信心,需驗證」)
- 用戶 ask「為什麼這檔有 3 個 standing」這類問題 → **先讀 standing_state 表,不要憑記憶**
