## 1. 重新審視自動化流程(2026-05-31 用戶提醒)

**用戶原話**:
> 「上線後記得提醒我要重新審一次自動化流程,
>   舊網站的自動化到後面沒辦法自己更新」

### 已知症狀(2026-06-01 凌晨深入確認)
- `kline.db` 最新只到 **2026-05-20**,5/21 ~ 5/31 共 **11 天沒新資料**
- **真因**:`tv_collect.mjs` 連 `data.tradingview.com/ping` 失敗,TradingView 內部 API
  session 失效
- Discord 通知機制**正常**:用戶 11 天都看到失敗訊息(`daily_supervisor` 健全)
- 排程本身**正常**:launchd 每天有觸發 `run_all.sh`,自動化骨架沒壞
- 卡的是 **5B 的資料源**(TradingView CDP)

### 為什麼會這樣
- `tv-cdp.plist`:`RunAtLoad=true, KeepAlive=false` → TradingView Desktop 一旦被關掉
  就不會自動重開
- `tv_collect.mjs` 連 TV API 失敗時**沒 retry 上限** → 5/31 跑 5+ 小時 retry,
  最後被用戶手動 kill
- 沒有「資料新鮮度」watchdog,只有「跑成功/失敗」狀態(失敗了但隔天又跑,
  狀態看起來「ok」卻沒新資料)

### 上線後要做的事
a) **資料新鮮度 watchdog**:檢查 `kline.db.MAX(date)` 跟今天差幾天,
   N 天沒新資料就告警(跟「跑成功/失敗」是不同訊號)
b) **`tv-cdp.plist` 改成定時 check + restart**(不只 login 時跑一次),
   確保 TradingView Desktop 一直在
c) **`tv_collect` 加 timeout 限制**(避免無限 retry 吃 CPU)
d) **或乾脆換掉 TradingView Desktop CDP**,改用其他 K 線來源
   (TWSE/TPEX 官方 API、yfinance、券商 API …)

### 用戶決策(2026-06-01)
- 「先不修,留給上線後重做」
- 開發階段繼續用 5/19 + 5/20 既有資料做 demo
